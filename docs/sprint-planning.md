## Sprint 1 (권장: 1주) — “블로그 MVP 동작 + 배포 파이프라인 정리”

### 목표
- placeholder 상태의 홈/포스트 상세를 **실제 콘텐츠 기반 블로그 MVP**로 전환
- 배포가 실제 빌드 산출물과 일치하도록 **배포 파이프라인을 정리**

### 데모 시나리오(스프린트 종료 시)
1) 홈(`/`)에 포스트 목록이 보인다(최소 2개)
2) 포스트 클릭 시 `/posts/[slug]`에서 내용이 렌더링된다
3) slug가 없는 경로는 404 처리된다
4) 배포된 URL에서 위 동작이 재현된다

## 범위

### In (P0 중심)
- S2.1 포스트 소스 형식/디렉터리 결정(문서화 + 샘플 포스트)
- S2.2 `/posts/[slug]` 실제 콘텐츠 렌더링 + 404
- S1.1 홈 포스트 리스트 렌더링 + 빈 상태
- S4.1 사이트 기본 메타데이터 정리(타이틀/설명)
- S5.1 배포 환경에서 빌드/런타임 확인(배포 워크플로 정합성 수정 포함)

### Out
- 태그/카테고리 페이지(S1.3)
- 분석/에러 로깅(S5.2~S5.3)
- 배경 끄기 토글(S3.3)

## 백로그(작업 항목)

아래는 “스토리 → 구현 작업” 단위로 쪼갠 백로그입니다.

### B1 (P0) 배포 파이프라인 정합성 수정 (S5.1)
- 작업
  - `deploy.yaml`의 `publish_dir`가 Next.js 산출물과 일치하도록 수정
  - GitHub Pages 유지 시: 정적 export 방식으로 전환(`out` publish 등)
  - 또는 호스팅 전환 시: 워크플로/문서 업데이트
- 완료 기준
  - Actions가 실제 결과물을 배포하고, 배포 URL에서 홈이 뜬다

### B2 (P0) 콘텐츠 디렉터리/포맷 결정 및 샘플 작성 (S2.1)
- 작업
  - `content/posts/` 등의 규칙 확정
  - frontmatter 스키마 확정(title/date/slug/description/tags)
  - 샘플 2개 작성
- 완료 기준
  - 빌드 시 샘플 포스트가 목록/상세에 노출된다

### B3 (P0) 포스트 로딩 유틸 구현 (S2.2/S2.3)
- 작업
  - slug 목록 생성
  - slug로 단일 포스트 조회
  - 정렬(최신순)
- 완료 기준
  - 잘못된 slug는 404

### B4 (P0) 홈 리스트 구현 (S1.1)
- 작업
  - 홈에서 목록 렌더링
  - 빈 상태 처리
- 완료 기준
  - 2개 이상 포스트 노출 + 링크 이동

### B5 (P0) 메타데이터 정리 (S4.1)
- 작업
  - `app/layout.tsx`의 `metadata`를 실제 블로그 정보로 업데이트
- 완료 기준
  - 브라우저 탭/OG 기본값이 의미 있는 값으로 표시

### B6 (P1) 배경 위 가독성 보완 (S3.1)
- 작업
  - 콘텐츠 영역에 오버레이/블러/컨테이너 스타일 적용
- 완료 기준
  - 홈/상세 본문 가독성이 유지된다
